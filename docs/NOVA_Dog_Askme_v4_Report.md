## 解决什么问题

工业机器人语音交互的 **6 个痛点**，Askme 逐一解决：

| 痛点 | 现状 | Askme 方案 |
|------|------|-----------|
| **延迟高** | 商业助手 2-5s 往返，机器人像在发呆 | 流式 TTS — 边生成边播放 |
| **不安全** | 误听一句就执行危险动作，没有确认机制 | 三级审批 — 危险操作必须人确认 |
| **会忘事** | 对话结束就忘了，每次从零开始 | 四层记忆 — 经验 → 反思 → 知识 |
| **不可控** | 云端黑盒，网络断了就哑了 | 本地 ASR/VAD/TTS — 离线也能听说 |
| **不可审** | 谁下的令？何时执行？无法追溯 | 全链路事件溯源 — 操作员身份追踪 |
| **不可复用** | 每个机器人重写一遍，技能无法共享 | 技能包插拔 — 新场景零代码扩展 |

## 五大创新

### 创新① 四层记忆 — 机器人会学习、会反思

> 不是 RAG，不是微调，是认知架构

| 层级 | 名称 | 机制 | 特点 |
|------|------|------|------|
| **L1** | 对话记忆 | 滑动窗口 | 最近 40 条消息，超量自动压缩为摘要 |
| **L2** | 会话记忆 | 中期摘要 | 每次对话结束生成 2-4 句摘要，按日期存储 |
| **L3** | 情景记忆 | 反思+知识 | 事件日志 + Ebbinghaus 遗忘曲线 + 自动反思 |
| **L4** | 向量记忆 | 长期语义 | 语义检索 (RAG)，懒加载，服务断不崩溃 |

#### "Aha" 时刻: 机器人每天巡检，第 7 天开始变聪明

- **Day 1**: 通用问答，不认识任何东西
- **Day 3**: 记住了哪些门容易卡（情景记忆）
- **Day 5**: 识别出固定时间出现的人员（模式反思）
- **Day 7**: 发现设备异常规律，主动预警（世界知识）

**L3 情景记忆核心机制**:
- 重要性评分: command(0.7) + 检测人(+0.5) + 危险(+0.6)
- 遗忘曲线: R = e^(-t/S), 每次回忆 S×2 (间隔效应)
- 反思触发: 累积重要性 ≥ 15 → LLM 提炼知识
- 5 类知识: **环境 · 实体 · 规律 · 交互 · 自省**

### 创新② 安全第一 — ESTOP 零延迟 + 三级审批

> 工业机器人的安全不是 feature，是底线

#### 意图路由 — 安全优先级

| 优先级 | 类型 | 示例 | 处理方式 |
|--------|------|------|----------|
| ① | **ESTOP** | "停！" "紧急停止" | 直达 safety 服务，不经 LLM |
| ② | **COMMAND** | "坐下" "站起来" | 确定性指令，不需要推理 |
| ③ | **TRIGGER** | "去厨房" "开始巡检" | 匹配 Skill，走任务编排 |
| ④ | **GENERAL** | "你好" "今天天气" | 才送到 LLM 自由对话 |

#### 工具三级安全

| 级别 | 操作 | 示例 |
|------|------|------|
| **normal** | 直接执行 | 查时间、读文件、看状态 |
| **dangerous** | 需语音确认 | 移动机械臂、网络搜索、执行命令 |
| **critical** | 最高警戒 (bypass: 免审批直达) | 紧急停止 |

#### 真实对话场景 — 安全审批

```
操作员: "把机械臂移到前面"
Askme:  识别为 robot_move (dangerous) → 需要确认
Thunder: "这是高风险操作: 移动机械臂到 (0, 100, 0)。请说'确认执行'继续，或'取消'放弃。"
操作员: "确认执行"
Thunder: "收到，正在移动机械臂...已到达目标位置。"
```

> 审批超时 30s 自动拒绝 · 确认词: 确认执行/继续执行/approve · 拒绝词: 取消/cancel

### 创新③ 流式语音 — 边生成边播放

**传统方案**: LLM 全部输出 → TTS 合成 → 播放 (等 8s+)
**Askme**: LLM 每出一句 → 立即 TTS → 立即播放

智能分句引擎 5 规则:
1. `。？！` → 立即分割 (强标点)
2. `；：` → 8字后分割 (中等标点)
3. `，` → 15字后分割 (逗号)
4. 60字 → 强制输出 (防卡顿)
5. 流结束 → 冲刷剩余内容

**效果**: 首句音频在 LLM 第一个句号后 ~500ms 开始播放

### 创新⑤ 语音 AI 基础设施 — 一次构建，赋能所有机器人

> Askme 不是一个产品，是一个平台

#### 三大核心价值

**硬件团队零 AI 门槛**
机器人公司的核心能力是硬件和运动控制，不应该再养一个语音 AI 团队。接入 Askme = 立即获得语音交互 + 安全审批 + 记忆学习 + 任务编排。

**场景快速复制**
新客户 ≠ 重新开发，新场景 ≠ 写新代码。通过管理后台调整: 机器人人格 · 语言风格 · 行为边界 · 技能包 · 安全策略 · 硬件适配。从签单到交付: **小时级，不是月级**。

**越用越值钱**
传统方案: 部署后能力固定。Askme: 部署后持续进化。四层记忆让每台机器人积累经验 — 同一条走廊走 100 遍后的机器人，比新出厂的机器人更懂这个环境。

#### 对穹沛意味着什么

| 方向 | 描述 | 亮点 |
|------|------|------|
| **巡检整机方案** | Thunder + Askme + LingTu 打包，语音控制导航 | 定制人格 + 巡检技能包 |
| **跨场景复制** | 同一平台换技能包，仓储/安防/接待快速复制 | 新场景交付周期: 天级 |
| **赋能集成商** | Askme 作为语音 AI 中间件，集成商赋能自己的机器人 | 按设备授权，持续收入 |
| **远程运维** | OTA 推送新技能/新策略，远程升级所有设备的能力 | 不用派人到现场 |

## 全链路 — "小穹，去厨房巡检"

完整的 12 步执行链路:

```
┌──────────────────────────────────────────────────────────────────┐
│  第一阶段: 语音输入 → 意图理解                                      │
├──────────────────────────────────────────────────────────────────┤
│  ① 用户说话        → "小穹，去厨房巡检"                             │
│  ② 唤醒 + 识别     → KWS → VAD → ASR → "去厨房巡检"               │
│  ③ 意图路由        → VOICE_TRIGGER → skill: patrol                │
│  ④ 边缘服务        → askme-edge 创建任务                           │
├──────────────────────────────────────────────────────────────────┤
│  第二阶段: 安全预检 → 任务执行                                      │
├──────────────────────────────────────────────────────────────────┤
│  ⑤ 安全预检        → dog-safety: estop? 电量? 黑名单?              │
│  ⑥ 任务编排        → arbiter: requested → approved    │
│  ⑦ 导航执行        → nav-gateway → LingTu gRPC                    │
│  ⑧ 遥测记录        → telemetry-hub: 事件 + 审计日志                 │
├──────────────────────────────────────────────────────────────────┤
│  第三阶段: 反馈 → 学习                                             │
├──────────────────────────────────────────────────────────────────┤
│  ⑨ 语音回复        → TTS: "好的，正在去厨房巡检"                     │
│  ⑩ 记忆更新        → L3: log(command), L1: 加入对话历史             │
│  ⑪ 后台反思        → 情景缓冲 → 反思 → 世界知识更新                  │
│  ⑫ 持续学习        → 下次再去厨房时，已经记住路线                     │
└──────────────────────────────────────────────────────────────────┘
```

## 6 个微服务 — 每个都可独立部署

> Docker Compose 一键启动 · 共享认证 · 事件溯源

| 服务 | 端口 | 角色 | 功能 |
|------|------|------|------|
| **askme-edge** | :5100 | 语音入口 | 接收语音 → 路由 → 聚合状态 → 返回回复 |
| **arbiter** | :5050 | 任务编排 | 全生命周期状态机 · 安全预检 · 审批流 · 审计 |
| **dog-safety** | :5070 | 安全守卫 | ESTOP · 维护模式 · 电量阈值 · 能力黑名单 |
| **dog-control** | :5080 | 动作执行 | 8 种 capability → Brainstem gRPC |
| **nav-gateway** | :5090 | 导航网关 | 6 种模式 → LingTu gRPC |
| **telemetry-hub** | :5060 | 遥测中心 | 事件流 · 审计日志 · 跨服务可观测 |

1. 写服务 — 标准模板
2. 加认证 — 共享一把钥匙
3. 部署 — Docker Compose 加一段就上线

## 数据对比

### 延迟瀑布对比

> 基于实测数据

| 处理阶段 | 传统串行方案 | Askme 并行+流式 |
|----------|-------------|----------------|
| 语音识别 ASR | 3.0s (云端 API) | 0.3-0.8s (端侧推理) |
| 记忆检索 | 3.0s (串行等待) | 0s (与 ASR 并行) |
| 意图路由 | 0.5s (LLM 判断) | <10ms (规则匹配) |
| LLM 首字节 | 5.0s (大模型) | 1.0s (快速语音模型) |
| TTS 合成 | 2.0s (全文缓冲) | 0.5s (首句流式) |
| **用户等待总计** | **~13.5 秒** | **~2-3 秒** |

#### 关键洞察: 双模型策略

- **思考模型**: 大参数模型 (深度推理，5s TTFT) — 适用于复杂问题
- **语音模型**: 小参数模型 (快速响应，1s TTFT) — 适用于日常对话

管理后台一键切换: 思考模式 ↔ 实时语音模式

#### 延迟优化公式

```
传统: ASR + 记忆 + 路由 + LLM + TTS (串行)
    = 3 + 3 + 0.5 + 5 + 2 = 13.5s

Askme: max(ASR, 记忆) + 路由 + 流式(LLM+TTS)
     = 0.8 + 0 + 0.01 + 1.0 + 0.5 = 2.3s
```

**节省 80%+ 延迟**

### 记忆增长曲线

> 基于 8h/天巡检场景推算

| 指标 | Day 0 | Day 1 | Day 3 | Day 7 | Day 30 |
|------|-------|-------|-------|-------|--------|
| L1 对话轮次 | 0 | 40 (滑动) | 40 (滑动) | 40 (滑动) | 40 (滑动) |
| L2 会话摘要 | 0 | 3 条 | 12 条 | 28 条 | 120 条 |
| L3 事件日志 | 0 | ~200 | ~600 | ~1,400 | ~6,000 |
| L3 反思次数 | 0 | 2-3 次 | 8-10 次 | 20+ 次 | 90+ 次 |
| L3 世界知识 | 0 | 5-10 条 | 25-40 条 | 60-100 条 | **200+ 条** |
| 知识类别覆盖 | 0/5 | 2/5 | 4/5 | **5/5** | **5/5** |

**推算依据**:
- 每次巡检指令: importance ~0.7
- 反思触发: 累积 importance ≥ 15 (~22 条指令)
- 反思冷却: 5 分钟，每天 ~3 次反思
- 每次反思产出: 3-8 条知识事实
- 遗忘曲线: S_init=1h, 访问×2, 上限 7 天

**这意味着**:
- Day 1: 机器人是新手，什么都要问
- Day 7: 认识了环境、人员、规律
- Day 30: **成为这个场地的"老员工"**

不需要重新训练 · 不需要手动标注 · 纯靠日常运行自动积累经验

### 安全响应时间

> ESTOP 路径 vs LLM 路径延迟对比

| 安全场景 | 云端语音助手 | Askme |
|----------|-------------|-------|
| ESTOP 关键词检测 | 无此概念 | **<10ms (规则匹配)** |
| ESTOP → 硬件停止 | 无此概念 | **<100ms (bypass LLM)** |
| 误听指令防护 | 直接执行 | **Intent Router 优先级拦截** |
| 危险操作确认 | 无确认机制 | **语音确认 (30s 超时自动拒绝)** |
| 离线急停 | 不可能 (云端) | **可用 (本地 ASR + 规则)** |
| 操作审计 | 无 | **telemetry-hub 全链路记录** |
| 安全策略远程更新 | 无 | **OTA 推送 policy 更新** |

#### ESTOP 路径 — 零 LLM 延迟

```
"停！" → ASR → IntentRouter (Tier 1) → dog-safety
跳过: LLM · 记忆检索 · Skill 匹配 · 工具调用
总延迟: ASR 端点检测 + <10ms = <1s
```

> Askme 的架构保证 ESTOP 不经过任何云服务。
>
> **合规说明**: ISO 13850 要求急停装置必须为物理设备 (按钮/拉绳/脚踏)。语音 ESTOP 作为**辅助快捷手段**，不替代物理急停按钮。实际部署需同时配备物理急停装置。

### 竞品详解

#### Picovoice (加拿大)
- **产品**: Porcupine (唤醒词) + Rhino (意图) + Cheetah (ASR) + Orca (TTS)
- **优势**: 全离线，隐私友好，Cheetah Fast 毫秒级延迟，支持 Jetson/ARM
- **劣势**: 只是语音原语组件库，没有 LLM/记忆/安全/技能编排，客户需自建上层应用
- **定价**: 免费开发 + 商用 $899-6,000+/月

#### 科大讯飞 (中国最大威胁)
- **产品**: 讯飞开放平台 + 讯飞星火大模型 + 机器人"语音背包"方案
- **优势**: 中文 ASR 准确率业界领先，已覆盖人形/仿生等多种机器人，全链路延迟 1.6s
- **劣势**: 云端依赖强，定价不透明，无工业级 ESTOP/安全审批框架
- **威胁**: **高** — 品牌强、客户粘性高，但不做垂直安全层

#### NVIDIA Riva
- **产品**: GPU 加速 ASR/TTS/NMT + Triton Inference Server
- **优势**: 延迟 <300ms (全行业最低)，高吞吐，企业级认证
- **劣势**: 需要 GPU 服务器 (V100/A100)，无记忆/安全框架，成本极高
- **定价**: NVIDIA AI Enterprise 许可 (企业询价)

#### 思必驰 / 云知声 (中国本地)
- **思必驰**: 全链路对话平台 + AI 芯片，专注车载/家居/机器人 OEM
- **云知声**: 蜂鸟芯片离线 ASR，车载后市场 >70%，医疗部署 60+ 家医院
- **共同劣势**: 专注消费级，无工业机器人安全/记忆能力

#### 机器人厂商自研
- **Unitree Go2**: 离线中文语音 + ChatGPT 云端对接，仅固定命令集，无安全审批
- **Boston Dynamics Spot**: 第三方 (Levatas) 集成 ChatGPT，非核心功能，$150k+ 硬件门槛
- **共同劣势**: 只解决自己产品，不可复用；功能固化，无记忆/学习能力

#### 开源方案 (Rasa / ROS2 Voice)
- **Rasa**: 强 NLU + 对话管理，完全本地可部署，但无 ASR/TTS/安全层
- **ROS2**: Vosk 离线 <200ms，但缺乏商业级支持，无安全认证
- **共同劣势**: 需要大量自研工作才能达到工业级

### Askme 的空白位置

```
高延迟 ─────────────────── 低延迟
│
│  Dialogflow          NVIDIA Riva
│  (企业云)            (数据中心GPU)
│
│ 科大讯飞 ◄─────────► Picovoice
│ (中文最强)           (隐私优先)
│
│ Rasa               Unitree Go2
│ (开源定制)          (量产消费)
│
│         ★ Askme 的目标位置
│  (LLM记忆 + 离线灵活 + 工业安全)
│
└───────────────────────────
  本地部署 ◄──────────► 云端服务
```

**Askme 的三角定位**: vs Picovoice 加 LLM+记忆 / vs 讯飞 减延迟加安全 / vs Rasa 加安全认证+商业支持

### 关键结论

1. **没有现成方案满足工业机器人语音控制的全部需求** — 记忆/安全/离线/技能的组合是空白
2. **科大讯飞是中文场景下最成熟的竞品** — 500+ 机器人客户，"语音背包"即插即用概念。需认真评估合作/竞争
3. **Unitree 语音很初级** — 四层记忆 + 安全确认是相对宇树的明显差异化
4. **记忆/学习是 Askme 独特优势** — 所有研究方案中没有一个提供类似四层记忆架构
5. **ISO 13850 合规** — 语音 ESTOP 只能作为辅助手段，必须配合物理急停按钮
